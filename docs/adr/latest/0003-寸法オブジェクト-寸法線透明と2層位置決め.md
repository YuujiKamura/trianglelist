# ADR 0003: 寸法値は「寸法線透明の寸法オブジェクト」として管理する

- 日付: 2026-06-10
- 状態: 実装済 — Phase 1 (式を common に、1742c96) → Phase 2a DXF (9993500) → 2b SFC (1dbf749) → 2c 画面+PDF (ab82caf) で全消費者が DimensionLayout に切替完了 (2026-06-10)。残: Triangle キャッシュ (dimPoint*) の引き剥がし = ADR 0002 Geometry 最終段階と合流 (起草経緯: yuuji 発話「本来は寸法線透明の寸法値として管理したほうが良いんだよな」2026-06-10。懸念「水平オフセットとか柔軟に移動させたりしづらくなるのかもしれないが」への解も本文に含む)
- 関連: ADR 0002 (当たり判定と3層分離)、insights #50 (dim-spacer-missing-in-dxf)

## Context

寸法線と寸法値の間のスペーサーが DXF に入っていない問題 (insight #50) の根本原因は、寸法というものを**線 (LINE) とテキスト (TEXT) のバラ**で管理していることにある。症状:

- スペーサーの実体が「画面描画時の `drawTextOnPath` の vOffset」(MyView.kt:663-665,841) にしかなく、DXF writer は素の辺上座標 + alignment コードだけを書く (DxfFileWriter.kt:159-161) — **配置の意図がデータに乗らない**
- `DimOnPath.textSpacer` は注入されるが誰も読まない死にフィールド
- 同じ「寸法」の知識が DimOnPath (offsetV/offsetH/旗揚げ HATAAGE)、MyView (textSpacer_、drawDigits)、DxfFileWriter (verticalDxf)、viewer (alignV 解釈) に分散

業界の確立形: AutoCAD の DIMENSION エンティティは寸法線・補助線・矢印・テキスト・隙間を**一体のオブジェクト + パラメータ群**で持つ:

- **DIMGAP**: 寸法線とテキストの隙間 (まさに欠けていたスペーサー。「DIMTAD=1 のとき寸法線からテキスト下端までの距離 = DIMGAP」と定義される)
- **DIMTAD**: テキストの上下配置 (0=中央 / 1=上 / 4=下 / **3=JIS 準拠**。JIS 専用値が規格化されている)
- **DIMSD1/DIMSD2・DIMSE1/DIMSE2**: 寸法線・補助線の非表示 (suppress) — **「寸法線透明の寸法値」は AutoCAD の標準パラメータで表現できる実在の概念**。三角形の辺自体が寸法対象である本アプリの図面は「寸法線・補助線を全 suppress した ALIGNED DIMENSION」と同型
- 出典: [Autodesk: D System Variables](https://help.autodesk.com/cloudhelp/2016/ENU/AutoCAD-LT-MAC/files/GUID-79A00CD5-A431-4460-8EA3-7F8707AEECA0.htm)、[CADDManager: AutoCAD Dimension Variables](https://www.caddmanager.com/CMB/2009/09/cad-standards-autocad-dimension-variables/)

### 懸念: 一体化すると柔軟な手動移動がしづらくなるのでは (yuuji 2026-06-10)

AutoCAD 自身がこれを **2 層**で解いている: デフォルト位置はパラメータ (DIMGAP/DIMTAD) から式で導出し、ユーザーがテキストをグリップで動かしたら text midpoint (DXF group code 11) を**手動オーバーライドとして記録**して式を上書きする (移動の挙動は DIMTMOVE: 0=寸法線ごと / 1=引出線付き / 2=自由移動)。「旗揚げ」は DIMTMOVE=1 の世界。

これは ADR 0002 で採用済みの 3 層の「覚える (PlacementOverrides)」と同型 — つまり**器は既にある**。一体化で柔軟性は失われない。むしろ現状は「override しか無い」(全テキストが生座標で、デフォルトの式が存在しない) 状態であり、式が無いからスペーサーも DXF に乗らず、自動配置も作れなかった。

### 既存手動操作の意味: 操作空間 = 接触回避の候補集合 (yuuji 2026-06-10)

「右寄せ左寄せ、左右旗揚げみたいなことを出来るようにしてあって、コレも実は寸法値同士が接触しないようにユーザーが調整するっていう事なんだが」

既存 UI の操作群 (offsetH の INRIGHT/INLEFT = 右寄せ/左寄せ、initPointsOuter の OUTERRIGHT/OUTERLEFT = 左右旗揚げ、DimOnPath.kt:68-79,114-123) の存在理由は**寸法値同士の接触回避**。これが意味すること:

- 手動操作の操作空間は、接触回避のための**離散的な配置候補の集合** {中央/右寄せ/左寄せ} × {内/外} × {定位置/右旗揚げ/左旗揚げ}
- ADR 0002 の LabelPlacer (自動の段階フォールバック) が試すべき候補は**この同じ集合** — ユーザーが手で 5 年やってきた回避手順を、当たり判定で評価して自動で選ぶだけ
- したがって自動と手動は**同じ語彙 (候補集合) を共有する**。ユーザー操作の記録は生座標でなく「左旗揚げを選んだ」という候補選択として残せ、自動はそれを尊重してスキップでき、将来は「自動提案 → ユーザー上書き」が同じ土俵で成立する

## Decision

**A. 内部モデルに寸法オブジェクト (仮称 DimensionLabel) を一級で立てる**

属性: アンカー辺 (対象 2 点) / 表示値 / **gapPaperMm** (DIMGAP 相当、紙単位) / 上下 (DIMTAD 相当: 内/外 = 既存 vertical 1/3) / 横オフセット (既存 offsetH = INRIGHT/INLEFT 等) / 旗揚げ状態 (DIMTMOVE 相当、既存 HATAAGE 計算を吸収) / footprint (ADR 0002 の LabelBox)。寸法線・補助線は描かない (透明) が、概念は ALIGNED DIMENSION と同型に保つ。

**B. 位置決めは 2 層: `position = デフォルト式(パラメータ) ⊕ 手動override`**

- デフォルト式: アンカー辺 + gap + 上下 + 横オフセットから閉形式で導出 (現 DimOnPath の式を吸収・整理)
- override は 2 種: **(a) 候補選択** (右寄せ/左寄せ/左右旗揚げ — 既存 UI の操作空間そのもの、式のパラメータ差し替えとして記録) / **(b) 自由座標** (グリップ移動相当)。ADR 0002 の PlacementOverrides がそのまま受け皿
- これにより「水平オフセットも自由移動も今まで通り」かつ「動かしていない寸法値は式が支配する」。自動配置 (LabelPlacer) は (a) と同じ候補集合を当たり判定で選ぶ — 自動と手動が同じ語彙

**C. 描画と書き出しは同じ寸法オブジェクトから導出する**

Android 画面 / desktop viewer / DXF / SFC / PDF の全 writer が、同じ「式 ⊕ override」の結果座標を消費する。スペーサー (gap) が画面にも DXF にも同じに乗る — insight #50 の根治。

**D. DXF 出力は当面 TEXT + LINE のまま** (CADWELL 互換を壊さない)。gap はデフォルト式の座標に織り込んで出す。DIMENSION エンティティでの出力は CADWELL の DIMENSION 解釈を実機検証してからの別判断。

## Consequences

- スペーサー欠落 (insight #50) が構造で直る。直すか直さないか (既存納品 DXF の見た目が変わる) は gap のデフォルト値で制御可能 (0 = 現状維持、>0 = 画面同等)
- 段階 3 (LabelPlacer、自動旗揚げ) の入力が「寸法オブジェクトの集合」として整い、ADR 0002 の配置エンジンが式の上に乗る
- 移行は段階的: まず common に寸法オブジェクトと式を立て、DimOnPath の知識を吸収 → 各 writer/描画を順次寄せる → Triangle から配置状態を引き剥がす (ADR 0002 の Geometry 最終段階と合流)
