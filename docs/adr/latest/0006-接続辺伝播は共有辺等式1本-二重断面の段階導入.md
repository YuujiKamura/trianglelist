# 0006: 接続辺の伝播は「共有辺の等式」1 本に縮約する / 二重断面接続の段階導入

- 日付: 2026-06-11
- 状態: 採用 → **同日改訂: 決定 1 の「1 hop コピー」を「辺プール (共有 index)」に置換** (下記 決定 1')
- 改訂の起点: user「リーナスが言っていた双方向連結リストの考え方を使うと、もっとシンプルに書けそうな気がする」→ 採用
- 起点: user 指示「接続辺をリスト上で数値変更した時に、接続されてる親または子の三角形にも変更が伝播するようにしろ。TriangleList クラスがそういうのをやってるはずだが、俺の実装は何だかよく理解ってないなりにナントカ形にしたものなので、適切な再設計のアイデアがあるなら考えてみろ」「親子の単純接続以外の、二重断面接続についてもそろそろサポートすることを考えろ」

## 現状分析 — アプリの伝播はなぜ複雑か

アプリ (TriangleList.kt) の伝播は **in-place のグラフ書き換え** で実現している:

- `resetFromParam` → `resetTriangles` (TriangleList.kt:537) → 親へ `resetByChild`、子へ `resetMyChild` (:590)
- 位置・角度・形状の 3 つが同時に変異するため、`childSide_` の付け替え、`nodeA` の再リンク
  (:551「再リンクしないと位置と角度が連動しない」)、連番/派生接続の分岐 (:598)、
  「ここがへん！」コメント (:568) の二重 reset 等、走査順と再リンクの帳尻合わせが本体になっている。

複雑さの根源は **「長さ (入力)」と「位置・角度 (導出値)」を同じ可変オブジェクトに同居させ、
導出値を差分更新している** こと。差分更新は走査順に依存し、リンク切れ・二重更新の温床になる。

## 決定 1 — web の伝播: 不変条件 1 本 + 1 hop コピー

web は毎描画 CSV から全再構築する (renderCsvToPrimitives)。位置・角度は**毎回ゼロから導出**
されるので、アプリが resetTriangles 群でやっている幾何の連鎖書き換えは **そもそも不要**。
state (行配列) に残る本質は不変条件 1 本だけ:

> 子の A 辺 ≡ 親の接続先辺 (conn 1=B, 2=C)。同一の物理辺だから常に等しい。

伝播は編集起点から 1 hop で完結する (再帰不要):

- A 辺を編集 → 親の B/C へ写す (上り 1 hop)。親の他の辺は変わらないのでそれ以上波及しない
- B/C 辺を編集 → その辺に接続している子の A へ写す (下り 1 hop)。子の他の辺は不変
- 孫以降の「位置のずれ」は wasm 再構築が自動で吸収する

実装 (初版): web/src/main.ts の `propagateEdgeValue` / `refreshCells`。→ 同日、決定 1' で置換し削除。

## 決定 1' — 辺プール: 伝播を「コード」から「構造」へ (同日改訂)

初版の 1 hop コピーは動くが、「同じ物理辺の長さが child.a と parent.b の 2 箇所に複製
されている」という表現の歪みをコードでパッチしている — リーナスの good taste の例
(単方向リスト削除の先頭特別扱いを間接ポインタで消す、mkirchner/linked-list-good-taste) で
言う 10 行版に相当する。表現を直す:

> 長さは `edges: string[]` プールに 1 個だけ。行は辺への index (`ea/eb/ec`) を持ち、
> **単純接続の子の A は親の B/C と同じ index を共有する**。

- 伝播は消滅 — 写す相手が存在しない (`setEdgeVal` 1 回で両者の実体が変わる)
- 上り/下り hop の分岐も消滅。表示の追従は `syncEdgeInputs` (同じ data-edge 属性の
  input を更新) だけ
- 接続の張り替え (parent/conn 編集) は `relinkEdgeA` — 単純接続なら親の辺 index を共有、
  独立/二重断面なら現値を複製して自分の辺に切り出す
- CSV 境界では従来どおり複製値に展開 (app 互換)。読込は「単純接続かつ数値が親辺と一致」
  の行だけ束ねるので、不一致の既存 CSV の描画は変えない
- これは計算幾何の DCEL (Doubly Connected Edge List) / half-edge / winged-edge の
  本質 1 点取り。面巡回などのフル機構は木構造には過剰なので持たない
  (van Rossen「half-edge considered harmful」の警告も踏まえた引き算)

不正値の据え置きは従来どおり三角不等式の共通関門 (redraw 内) が担う。

出典: mkirchner/linked-list-good-taste (Linus Torvalds TED 2016 の解説) /
cs418.cs.illinois.edu の half-edge 解説 / Max McGuire "The Half-Edge Data Structure" (2000)

## 決定 2 — アプリ側への再設計提案 (将来)

アプリも同じ縮約が可能: 「長さの行リスト (SoT) → 全再構築」のパスは CSV round-trip として
既に存在する (CsvLoader.buildTriangle → trilist.add(Triangle(ptri, cp, B, C)))。
resetTriangles / resetMyChild / resetByParent / childSide_ 帳尻合わせは、
「行リストを書き換えて先頭から組み直す」に置き換えれば消せる。性能が要る時だけ
差分更新を最適化として戻す (この規模ではまず不要 — insight #55 の draw call 律速と同じ理屈)。
ただしアプリ改修はリスクが大きいので、web で運用実績を積んでから別 ADR で判断する。

## 決定 3 — 二重断面接続の段階導入 (web)

二重断面 = 単純接続 (子 A = 親辺長、Center 揃え) ではない接続。データ上は完全形式 CSV の
`ConnParam(side, type, lcr, lenA)` (列 17-19、CsvLoader.kt TriangleColumn) で表現され、
子 A ≠ 親辺長のまま L/C/R に寄せて配置される。アプリの「二重」FAB (MainActivity.kt:1322) は
`rotateCurrentTriLCR` (TriangleList.kt:488) で LCR を巡回させる。

導入順 (引き算: 一度に全部やらない) — **2026-06-11 同日、①〜③実装完了 (commit 5912501)**:

1. **読み・保持 (済)**: web の Row.extras は列 6 以降を生のまま round-trip するので、
   完全形式 CSV を開いても二重断面情報は壊れない。
2. **描画 (済)**: 写像を common/ConnCode.kt に一本化 (コード⇔ConnParam、SoT は
   setCParamFromParentBC と同表) し、WebCsvReader と app の CsvLoader の双方が使う。
   web が書く 6 列 CSV (コード 3-10) を app も読める。
3. **操作 UX (済)**: UI はアプリの 11 択 spinner を**3 列に分解** —
   「接続辺 (親のB辺/C辺)」×「形態 (辺共有/二重断面/フロート)」×「起点 (右/中央/左)」
   (user 設計: 11 択は事情を知らないと判らない、3 列は ConnParam の side×type×lcr と同型)。
   新規行に「独立」は無い — 最初の 1 個だけ自動で独立 (スマホ版同様、図面に 1 リスト)。
   「二重」FAB は conn コードの lcr 巡回 (右→中央→左) + redraw だけ —
   アプリの rotateCurrentTriLCR の in-place 連鎖が、全再構築+辺プール設計では
   値の書き換え 1 行になる。
4. **検証 (一部済/残)**: CP e2e (C辺二重断面・右起点 追加→capture 目視、FAB 巡回 5→8→
   中央へ移動を目視) 済。**残: アプリで作った二重断面入り実 CSV を golden に app/web
   描画一致を見る** (完全形式 28 列の読みは列 17-19 のみ対応、レイアウト列は未対応のまま)。

## 付記 — Kotlin 2.2.10 の wasm 成果物非互換 (2026-06-11 同日に踏んだ)

ツールチェーン更新 (commit b7f9baf) で 2 点が変わり「三角形が出なくなった」:
1. 出力名が `<project>-wasm-js` → `<project>` に変更 → `outputModuleName` で旧名固定。
2. uninstantiated.mjs の .wasm fetch が document base 相対 → モジュール相対
   (`new URL('./x.wasm', import.meta.url)`) に変更 → 旧配置 (web/public 直下) だと
   /wasm/x.wasm への fetch に Vite の SPA フォールバックが index.html を返し、
   WebAssembly.instantiateStreaming が MIME エラーで即死 (status「loading wasm...」凍結)。
   sync-wasm.mjs が .wasm を .mjs の隣 (web/wasm/) にも置くよう修正。
